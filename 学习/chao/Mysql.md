# Mysql



## 慢SQL



### 如何排查

1. 先看CPU和内存使用率，是不是系统资源不够导致阻塞的慢查询，比如并发过高，阻塞住了，或者连接池耗尽阻塞，或者锁竞争
2. 一般都有慢查询日志，接入告警的话，能精准的看到是哪条sql执行时间不符合预期
3. 没接入的话可以通过调用trace确定sql耗时



### 如何分析

**EXPLAIN**

“慢 SQL 本质上是扫描了不该扫描的数据量。”

关注项目

- type（是否全表扫描 ALL）

  system/const(唯一索引命中) > ref(普通索引命中) > range(索引范围) > index(索引全表，覆盖索引) > ALL(扫全表)

- key（有没有走索引）

- rows（扫描行数）



### 定位问题

**1. 索引问题**

- 没建索引
- 索引失效（函数、隐式类型转换、like ‘%xx’）
- 联合索引未命中最左前缀



**2. SQL 写法问题**

- select *（回表严重）
- 大范围扫描（无分页或分页过大）



**3. 数据问题**

- 表数据量过大（千万级+）



### 优化

**SQL 层**

- 使用覆盖索引，减少回表
- 数据量大时，尽量不使用in，like
- 分页优化（避免分页太大）



**索引层**

- 建合适的联合索引
- 避免索引失效
- 使用前缀索引（字符串）



**架构层**

- 分库分表（数据量太大，单表承载不了）
- 引入缓存（Redis / 本地缓存）



### 验证

上线前压测

测试环境数据量少，并发低，可能看不出问题，线上压测一下，很多问题可能会暴露出来





## 索引失效

本质：查询条件破坏了索引的**有序性**或**可定位性**，使优化器无法用 B+Tree 做快速查找，只能退化为扫描。



### **对索引列做“变换”**

- 函数操作

  where date(create_time) = '2024-01-01'

- 表达式 / 运算

  where price * 1.1 > 100

- 显式类型转换

  where cast(id as char) = '123'

👉 机制：

原本索引里是“原始值的顺序”，你比较的是“计算后的值”，顺序关系失效 → 无法走索引。



### **不满足最左前缀（联合索引）**

索引 (a, b, c)：

- 跳过最左列

  where b = 1

- 中间断裂

  where a = 1 and c = 3

👉 机制：

联合索引是按 (a→b→c) 排序的，必须从左连续匹配；断了，后面的列都用不上。



### **范围查询截断**

```sql
where a = 1 and b > 10 and c = 5
```

👉 机制：

- a：精确定位
- b：范围扫描
- 到 c：已经进入区间，无法继续精确匹配

👉 结论：**范围右侧列全部失效**



### **LIKE 使用不当**

- %abc（前缀不确定） → 失效
- abc%（前缀确定） → 可用索引

👉 机制：

B+Tree 只能从左开始匹配，不能从中间定位。



### **索引选择性太差 / 优化器放弃**

```sql
where gender = 'male'
```

如果表里 90% 都是 male：

👉 优化器可能直接走全表扫描（type=ALL）

原因：

> 用索引还要回表，不如直接扫一遍更便宜。
